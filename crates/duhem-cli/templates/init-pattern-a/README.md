# `{{NAME}}`

A Duhem Verification Definition skeleton, generated by
`duhem init`. Out of the box this verifies that
`https://example.com` renders its canonical heading — a known-
good baseline you mutate toward your real workload.

- Criteria prose: [`criteria.md`](criteria.md)
- Verification Definition: [`duhem.yml`](duhem.yml)

## What to do next

1. Edit `criteria.md` to express the real commitments of your
   feature. One commitment per `AC-N` block; keep the prose
   intent-bearing.
2. Edit `duhem.yml` to mechanically verify each criterion: one
   or more `checks:` per criterion, each with `steps:` (actions
   that exercise the real artifact) and `assertions:` (boolean
   expressions over `$steps.<id>.outputs.*`). The action
   catalog lives in `docs/duhem-spec.md` §10.5.
3. Re-validate and re-run after each edit.

## Required inputs

The skeleton declares one input with a default:

| Input      | Type   | Default                |
| ---------- | ------ | ---------------------- |
| `base_url` | string | `https://example.com`  |

Add `inputs:` entries (and `--inputs key=value` flags) as your
verification grows. See `docs/duhem-spec.md` §7.5 for the input
catalog rules.

## Environment prerequisites

- Outbound HTTPS to `https://example.com` (or whichever URL
  your edited `duhem.yml` targets).
- Playwright Chromium installed:
  ```sh
  npx playwright install chromium
  ```

## Running the verification

Validate the file structure:

```sh
duhem validate {{PATH}}/duhem.yml
```

Run end-to-end (launches Playwright, hits the real URL):

```sh
duhem run {{PATH}}/duhem.yml
```

Override the default input:

```sh
duhem run {{PATH}}/duhem.yml --inputs base_url=https://example.org
```

Iterate on a single criterion while editing (`--filter` syntax
in `docs/duhem-spec.md` §13):

```sh
duhem run {{PATH}}/duhem.yml --filter AC-1
```
